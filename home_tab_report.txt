Technical Description: HomeTab Frontend Implementation

1. Overview and Architectural Role

The `HomeTab` widget, defined in `lib/screens/home/views/home_tab.dart`, implements the main dashboard-style home page of the Voyant Flutter app. It is responsible for:
- Displaying a personalized greeting using user profile data stored in Cloud Firestore.
- Showing the user’s XP, level, and progress toward the next level in real time.
- Fetching and displaying usage statistics (quest count, trip count, active trip) from a backend REST API.
- Providing navigation entry points to the referral screen, settings screen, profile screen, and trips view.

Architecturally, `HomeTab` is a `StatefulWidget` that combines:
- A reactive Firestore stream for user document updates.
- An imperative HTTP call to a backend API for summary statistics.
- Local state to manage loading indicators and backend data.
- A custom animated background component to give the page a distinct look.

2. Widget Structure and State Model

The `HomeTab` class is a `StatefulWidget` that accepts a single callback:
- `final VoidCallback onTripsTap;` – a function provided by the parent widget to handle navigation to the Trips section when the Trips stat card is tapped.

The associated state class `_HomeTabState` defines:
- `static const String baseUrl`: The base URL of the backend API (`https://api-cbmysz2x4a-uc.a.run.app/api`).
- `Map<String, dynamic>? stats`: Holds the JSON-decoded statistics payload returned from the backend (e.g., quest count, trip count, active trip details).
- `bool isLoading`: Indicates whether the app is currently fetching statistics from the backend. This controls loading spinners and conditional rendering.

On initialization (`initState`), the state calls `_loadStats()` to fetch backend statistics as soon as the widget is inserted into the tree.

3. Authentication Integration

The widget uses Firebase Authentication to authenticate backend requests and to identify the current user in Firestore:

- `_getToken()` is an async helper that calls `FirebaseAuth.instance.currentUser?.getIdToken()` and returns the Firebase ID token as a `String?`.
- This ID token is included as a Bearer token in the `Authorization` header of HTTP requests to the backend.
- The current user object is retrieved in `build` via `FirebaseAuth.instance.currentUser`, and its `uid` is used to subscribe to the corresponding Firestore document.

4. Loading Backend Statistics via HTTP

Backend home statistics are fetched by the `_loadStats()` method:

- It is an asynchronous method called from `initState`.
- Execution flow:
  1. Obtain a Firebase ID token via `_getToken()`.
  2. Issue an HTTP GET request to `GET {baseUrl}/stats/home` using the `http` package:
     - `http.get(Uri.parse('$baseUrl/stats/home'), headers: {'Authorization': 'Bearer $token'},)`.
  3. If the response has status code 200:
     - Decode the JSON body using `jsonDecode(response.body)`.
     - Store the resulting `Map<String, dynamic>` in the `stats` field.
     - Set `isLoading = false` inside `setState` to trigger a UI update.
  4. If the response status is not 200 or an exception occurs:
     - Log the error via `debugPrint` (for debugging visibility).
     - Set `isLoading = false` so that the UI stops displaying the loading indicator.

This pattern cleanly separates backend statistics from Firestore-sourced user data: Firestore is used for user-specific real-time XP and profile info, while the API provides broader usage metrics linked to the authenticated account.

5. XP, Level, and Progress Computation

The page shows the user’s XP, current level, and progress toward the next level. These values are derived from an integer `totalXP` field stored on the Firestore user document.

The following helper methods encapsulate the level logic:
- `_getLevel(int xp) => (xp / 1000).floor();`
  - Interprets every 1000 XP as one level.
  - For example, 0–999 XP → level 0, 1000–1999 XP → level 1, etc.
- `_getXPForNextLevel(int xp) => 1000 - (xp % 1000);`
  - Computes how much XP is needed to reach the next level by subtracting the remainder of `xp` modulo 1000 from 1000.
- `_getLevelProgress(int xp) => (xp % 1000) / 1000;`
  - Computes a normalized value between 0 and 1 indicating progress within the current level, suitable for a `LinearProgressIndicator`.

These helper functions allow the UI to remain simple and declarative, with all numeric transformations encapsulated in reusable methods.

6. Displaying the User’s Name Robustly

The `_displayFirstName(Map<String, dynamic>? userData)` method is responsible for computing a user-friendly display name for the greeting:

- It first attempts to read `firstName` from the `userData` map, trimming whitespace.
- If `firstName` is non-null and non-empty, it returns this value.
- Otherwise, it falls back to the `username` field:
  - If `username` is null or empty, it returns the generic label "Explorer".
  - If `username` is present, it splits the string on whitespace and returns only the first token, providing a first-name-like display even if the stored username is a full name.

This strategy ensures the greeting line always has a sensible value, even in the presence of partially filled profile data.

7. Real-Time User Data with Firestore StreamBuilder

The core of the page’s reactivity comes from a `StreamBuilder<DocumentSnapshot>` wrapping the main content:

- The stream listened to is:
  `FirebaseFirestore.instance.collection('users').doc(user?.uid).snapshots()`
- This subscribes to real-time updates of the user’s Firestore document.
- Inside the `builder` callback:
  - The snapshot’s data is cast to `Map<String, dynamic>?`:
    `final userData = snapshot.data?.data() as Map<String, dynamic>?;`
  - The user’s display name is derived via `_displayFirstName(userData)`.
  - XP and level-related values are computed from `userData?['totalXP']` (defaulting to 0 when null):
    - `final totalXP = (userData?['totalXP'] ?? 0) as int;`
    - `final level = _getLevel(totalXP);`
    - `final xpToNext = _getXPForNextLevel(totalXP);`
    - `final levelProgress = _getLevelProgress(totalXP);`

Because this is a `StreamBuilder`, any update to `totalXP` or other fields on the Firestore user document is immediately reflected in the UI without manual refresh. This is especially useful if XP is updated from other parts of the app (e.g., completing quests).

8. Overall Layout Composition (build Method)

The `build` method creates a `Scaffold` with a rich, custom background and layers several sections vertically using a `Column` inside a `SingleChildScrollView`.

8.1. Scaffold and Background

- `Scaffold` provides the page structure.
- The body is wrapped in an `AnimatedGradientBackground` widget (custom component from `voyant/widgets/animated_gradient_background.dart`), which renders an animated gradient effect behind the entire content.
- `SafeArea` is used to avoid system UI intrusions (e.g., notch and status bar).

8.2. Scrollable Content

- A `SingleChildScrollView` with `padding: const EdgeInsets.all(20)` ensures that content can scroll vertically on smaller devices and maintains consistent padding.
- Inside, a `Column` lays out the subsections:
  1. Header row (greeting and icons).
  2. XP and level card.
  3. Statistics row from backend.
  4. Active trip section.

9. Header Section: Greeting and Navigation Icons

The header section is implemented as a `Row` with `mainAxisAlignment: MainAxisAlignment.spaceBetween`:

- Left side: a `Column` with two `Text` widgets:
  - "Welcome back," in small, grey text.
  - The user’s name, computed via `_displayFirstName`, in large, bold, white text.

- Right side: a `Row` of three interactive icons:
  1. **Referral Icon** (`Icons.card_giftcard`):
     - Tapping it uses `Navigator.push` with a `MaterialPageRoute` to navigate to `ReferScreenView()` from `voyant/screens/referral_system/refer_screen.dart`.
  2. **Settings Icon** (`Icons.settings`):
     - Similarly navigates to `SettingsScreen()`.
  3. **Profile Icon**:
     - Implemented as a circular `Container` with a gradient background and border, containing an `Icons.person` icon.
     - Wrapped in a `GestureDetector` that navigates to `ProfileScreen()` on tap.

This header consolidates identity (greeting) and quick navigation to important user-centric sections.

10. XP and Level Card

The XP and level information is encapsulated in a stylized `Container`:

- Layout and styling:
  - `width: double.infinity` and `padding: const EdgeInsets.all(20)`.
  - Background is a `LinearGradient` from a deep purple to a dark tone, with rounded corners and a purple border.
- Internal composition:
  - A `Row` at the top with:
    - Left: label "Explorer Level" in semi-transparent white.
    - Right: a small badge-like `Container` showing the current level (`LVL $level`) on a solid purple background.
  - Large XP text: `'$totalXP XP'` in bold white, 32pt font.
  - A `LinearProgressIndicator` wrapped in a `ClipRRect` with rounded corners, using `levelProgress` as its `value` and a purple `valueColor`.
  - A small caption: `'$xpToNext XP to Level ${level + 1}'` in semi-transparent white, indicating the remaining XP required for the next level.

All numeric values here are derived from Firestore data and updated in real time.

11. Statistics Row from Backend

Below the XP card, the page shows a row of statistic cards. This section is controlled by `isLoading` and the `stats` map filled by `_loadStats()`:

- If `isLoading` is true:
  - A centered `CircularProgressIndicator` with a purple color is displayed.
- Otherwise, a `Row` with three stat cards is built:
  1. **Quests Card**
     - Icon: `Icons.flag`, orange color.
     - Value: `stats?['questCount'] ?? 0`, interpolated into a string.
     - Label: "Quests".
     - No `onTap` callback, purely informational.
  2. **Trips Card**
     - Icon: `Icons.luggage`, purple color.
     - Value: `stats?['tripCount'] ?? 0`.
     - Label: "Trips".
     - `onTap: widget.onTripsTap`, which delegates navigation logic to the parent widget. This allows the home page to instruct the parent (likely a tab controller or navigator) to switch to the trips tab.
  3. **Badges Card**
     - Icon: `Icons.military_tech`, amber color.
     - Value: currently hard-coded as `'0'`.
     - Label: "Badges".
     - No tap handler; placeholder for future feature expansion.

11.1. Reusable Stat Card Builder

The stat cards are built using a private helper method `_buildStatCard`:

```dart
Widget _buildStatCard(
  IconData icon,
  Color color,
  String value,
  String label,
  VoidCallback? onTap,
)
```

- The method creates a `Container` with:
  - Padding and rounded corners.
  - A semi-transparent purple background and dark border.
- Inside, a `Column` centers the icon, value, and label vertically.
- If `onTap` is not null:
  - The card is wrapped in a `GestureDetector` for tap handling.
  - The result is wrapped in an `Expanded` so cards share available horizontal space evenly.
- If `onTap` is null:
  - Only the `Expanded` wrapper is applied.

This helper encapsulates all styling and layout logic for stat cards, reducing duplication and making it easy to add more cards in the future.

12. Active Trip Section

At the bottom of the page, the current active trip (if any) is displayed, based on the `stats` data returned by the backend:

- A heading text: "Active Trip" in bold white.
- After a small vertical gap, the content is conditionally rendered:
  - If `isLoading` is true: an empty `SizedBox` is shown (no trip info while loading).
  - Else if `stats?['activeTrip'] == null`: a simple `Text('No active trip')` in `Colors.white54` is displayed.
  - Else: a detailed card for the active trip is rendered.

12.1. Active Trip Card Layout

When `stats['activeTrip']` is not null, it is expected to contain fields such as:
- `name`: the trip name.
- `completedQuests`: the number of quests completed in this trip.
- `totalQuests`: the total number of quests in this trip.

The card is a `Container` with:
- Consistent padding and a dark background color (`Color(0xFF12121A)`).
- Rounded corners and a purple border.

Inside, a `Column` structures the content:
- Top `Row`:
  - Left: trip name in bold white text.
  - Right: a small green badge labeled "Active" indicating that this trip is ongoing.
- A `LinearProgressIndicator` visualizing quest completion:
  - `value` is computed as:
    - `0` if `totalQuests` is 0 (to avoid division by zero).
    - Otherwise, `completedQuests / totalQuests`.
  - Uses `Colors.green` as the progress color and a light translucent background.
- A caption line showing the ratio: `'{completedQuests}/{totalQuests} quests completed'`, styled with grey text.

All values are pulled directly from the `stats!['activeTrip']` map, ensuring that any backend updates to active trip data are reflected when `_loadStats` is next executed (e.g., on app open or refresh).

13. Visual Design Notes

Several consistent design choices give the page a cohesive look:
- Background uses an animated gradient from a custom widget, distinguishing this as a primary screen.
- Main accent color is purple (`0xFFB020DD`), used in:
  - Icons (referral, settings, some stats).
  - Borders and gradients on containers.
  - Progress indicators and badges.
- Secondary colors are darker purples and near-black backgrounds to provide contrast while maintaining a unified theme.
- Text colors range from pure white for primary information to greys for secondary hints.

14. Data Flow Summary

The `HomeTab` page’s data flow can be summarized as follows:

- On widget initialization:
  - `initState` triggers `_loadStats()`.
  - `_loadStats` obtains a Firebase ID token and calls the backend `/stats/home` endpoint.
  - `stats` and `isLoading` are updated based on the HTTP response.

- In parallel, the `StreamBuilder` subscribes to the `users/{uid}` Firestore document:
  - Any change to `totalXP`, `firstName`, `username`, etc., automatically rebuilds the UI with the new values.

- As a result:
  - XP and level card reflect real-time XP changes across the app.
  - Statistics row and active trip card show server-side aggregates driven by the authenticated user.

15. Navigation Responsibilities

The home tab also serves as a navigation hub via its header and statistics cards:

- Header icons:
  - Referral icon → `ReferScreenView()`.
  - Settings icon → `SettingsScreen()`.
  - Profile circular avatar → `ProfileScreen()`.
- Trips stat card:
  - Invokes the `onTripsTap` callback provided by the parent widget rather than directly pushing a route. This separation allows the parent (e.g., a bottom navigation controller) to implement navigation logic in a centralized manner.

Overall, the `HomeTab` frontend page combines Flutter UI composition, Firestore real-time streams, Firebase Authentication, and REST API integration to build a responsive home dashboard that reflects both user-centric progress (XP/level) and broader app usage (quests, trips, active trip).
